# Проверка переносимости migration-документа на другие Electron-приложения

Дата проверки: 2026-02-23  
Проверяемый документ: `docs/electron-to-tauri-migration-requirements.md`

## Итог

Документ **можно использовать как базовый runbook для миграции другого Electron-приложения**, если у приложения сопоставимая архитектура (renderer/main/preload, IPC, локальное хранение, desktop-интеграции).

Статус: `условно годен` (готов как baseline, требует адаптации под конкретный app profile).

## Что уже переносимо без изменений

1. Фазная стратегия миграции (инвентаризация -> adapter -> Tauri host -> sidecar bridge -> перенос persistence -> cutover).
1. Требования к UI-абстракции платформы.
1. Требования к IPC-контракту и разделению ownership между host и sidecar.
1. Security-подход (host-first валидация путей/URL/capabilities).
1. Критерии готовности (DoD) и release-гейты.

## Что требует адаптации для другого приложения

1. Набор и имена событий IPC.
1. Схема данных и формат legacy-хранилищ.
1. Список Electron-only возможностей (tray, notifications, global shortcuts, browser windows).
1. Packaging pipeline и целевые артефакты.
1. Матрица tool-паритета (особенно при native Node зависимостях).

## Риски повторного использования без адаптации

1. Ложное ощущение parity при несовпадении IPC-событий.
1. Ошибки миграции данных при отличной схеме/путях.
1. Потеря функционала в Electron-only фичах.
1. Недостаточная security-модель, если оставить проверки только на UI-слое.

## Минимум обязательной адаптации перед стартом

1. Заполнить inventory для целевого проекта.
1. Составить конкретный IPC mapping `old -> new`.
1. Сформировать tool parity matrix `host-native | sidecar | deferred`.
1. Подготовить data migration plan + rollback plan.
1. Зафиксировать release smoke-тесты на целевых ОС.

## Вывод

Документ пригоден как практическая основа для миграции других Electron-проектов, но должен применяться вместе с обязательной адаптацией под архитектуру и ограничения конкретного приложения.
